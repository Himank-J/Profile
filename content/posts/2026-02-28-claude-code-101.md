## The Core Mental Model

Most LLM workflows look like this:

```
Prompt → Model → Output
```

Claude Code looks like this:

```
User Request
      ↓
Parent Agent (Orchestrator)
      ↓
Planning (if plan mode enabled)
      ↓
Delegation (Sub-agents / Skills / MCP)
      ↓
Tool Execution
      ↓
Aggregation
      ↓
Validated Output
```

The key difference:

Claude Code is an **orchestrated system**, not a single forward-pass model interaction.

## Memory Architecture

Claude Code separates configuration and memory into:

* Global (~/.claude/)
* Project (.claude/)

This is not arbitrary. It solves a real architectural problem.

### Global Layer — ~/.claude/

**What it is** - User-level configuration and preferences that apply everywhere.

**Why it exists** - 

If every project redefined Language preferences, Code formatting rules, Testing standards, Naming conventions, you’d create repetition and drift.

The global layer allows:

* Personal AI coding identity
* Persistent behavioral defaults
* Cross-project consistency

Instead of re-prompting every project, you define this once.

### Project Layer — .claude/

**What it is** - Repo-specific configuration.

**Why it exists**
Engineering constraints differ per repo:

* One project may be microservices, another may be a monolith.
* One may enforce Clean Architecture, another may be experimental.

You don’t want global preferences overriding project-specific architecture.

So Claude Code isolates:

```
Global Identity
      +
Project Constraints
```

This enables safe collaboration.

### CLAUDE.md — The most important file for claude

If you’ve used Claude Code and felt like it wasn’t that great, it’s probably not the tool — it’s the setup. Most likely, you’re either not creating this file or not using it the right way.

This file acts as:

* High-level instructions
* Architectural guardrails
* Behavioral constraints

But more importantly: it reduces ambiguity, LLMs struggle when rules are implicit.

CLAUDE.md makes:

* Architecture explicit
* Constraints explicit
* Style expectations explicit

Best practice: always use this file when working on complex or multi-step problems.

When creating it, clearly define the following to get the most out of Claude Code:

* **Background on the task** – Provide relevant context so the model understands the bigger picture.
* **Objective / Goal** – Clearly state what success looks like.
* **Inputs & Outputs** – Specify what goes in and what should come out.
* **Guidelines** – Outline constraints, rules, or preferences to follow.
* **Warnings / Exceptions (optional)** – Call out edge cases, limitations, or things to avoid.
* **References (optional)** – Include external links, images, or documentation for additional context.

The more structured and explicit this file is, the better the results will be. That dramatically reduces hallucination and architectural drift.

## Agent Architecture

Claude Code uses a parent agent + sub-agent model. This mirrors distributed systems thinking.

### Parent Agent (Orchestrator)

**What it does**

* Understands user intent
* Decides execution strategy
* Determines if planning is needed
* Spawns sub-agents
* Loads skills
* Calls tools
* Merges outputs

**Why it exists**

You don’t want one giant agent handling everything with full context.
That leads to:

* Token explosion
* No specialization
* Slow reasoning
* Increased hallucination risk

Instead you separate orchestration from execution.

### Sub-Agents

Sub-agents are scoped workers.

They:

* Operate with limited context
* Have limited tool permissions
* Can use cheaper models (token savings)
* Run in parallel (optimised workflows)

**Why limit them?**

Because:

* Large context windows degrade reasoning clarity.
* If you give an agent 200 files, its attention dilutes.

Sub-agents instead get only what they need. This improves **precision, cost, speed**.

## Skills System

Skills are modular capabilities that are **loaded only when required**.

**Why Skills Exist**
Without skills, every complex task requires:

* Large prompt instructions
* Repeated behavioral explanations
* High token usage

Skills act as **Reusable procedural modules**.

```
# Skills directory setup

.claude/
  skills/
    refactor-python/
      skill.md
      scripts/
        transform.py (any script)
```

**What is skill.md?**

This defines:

* Name
* Description
* Goal
* Tools required
* Instructions
* Invocation logic

It tells the parent agent:
“When to use this capability.”

### scripts/ directory

This is critical — and often misunderstood.

**What it is** - A directory that contains executable code used by the skill.

**Why it exists**
LLMs are great at reasoning, but they are not deterministic executors.

If you want:

* Reliable code transformations
* Structured file manipulations
* Schema migrations
* Static analysis
* Deterministic rewrites

You don’t want pure model inference. You want real code.

So:

* The skill defines the logic
* The script executes deterministically

*Example Use Case*

Skill: “Auto-add type hints to Python files”

Instead of LLM rewriting every file manually

You:

* Write a script using lib2to3 or AST parsing
* The model invokes that script
* Script performs reliable transformation

This improves:

* Determinism
* Safety
* Repeatability

And **reduces token consumption significantly**

Skills combine: LLM reasoning + real code execution

## Plan Mode

Plan mode introduces a separation between, Thinking and Acting.

Enabling Plan Mode shifts the agent’s focus from execution to structured thinking. Instead of jumping straight into implementation, it prioritizes outlining the steps first.

This makes the agent more deliberate and cautious, while also improving clarity by cleanly separating **“what needs to be done”** from **“how it should be done.”**

**Why this matters:**
Large-scale refactors require:

* Dependency awareness
* Migration sequencing
* Risk analysis

Plan mode turns Claude from reactive to deliberative.

## Model Context Protocol (MCP)

MCP enables integration with external systems.

Think of it as:

A standardized interface between Claude and tools.

Examples: GitHub, Databases, Search engines, CI systems

**Why MCP Exists**

* You don’t want: Prompt-based imitation of external systems.
* **You want**: Structured tool access.

MCP provides Reliable, structured, composable integrations.

## Agent Teams — Structured Engineering Simulation

Agent teams simulate real engineering pipelines.

Instead of one agent doing everything sequentially.

You define agents for:

* Research
* Architecture
* Implementation
* Review
* QA

This:

* Improves structure
* Reduces logical errors
* Mimics real software teams
* Enables parallelism

It’s AI-driven SDLC.

You can create custom agents by defining `agent.md` file within "agents" directory. By defining agent role and access to tools, claude can spawn if when needed. For example - creating a "scrap_website.md" within agents directory can define an agent for scarping website.

**Enable agent teams**

Agent teams are disabled by default. Enable them by setting the
CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS environment variable to 1, either in your shell environment or through settings.json:

```
settings.json
{
"env": {
   "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": 1
   }
}
```

**Note** - For creating custom customs you dont need to enable agent teams mode.

### Sub-Agents vs Agent Teams

Both agent teams and subagents let you parallelize work — but they serve different purposes. The right choice depends on whether your workers need to communicate with each other.

* Subagents

  * Context: Each subagent has its own context window, but results are returned to the main agent.
  * Communication: They report back only to the main agent.
  * Coordination: The main agent manages and delegates all work.
  * Best for: Focused, independent tasks where only the final output matters.
  * Token cost: Lower — results are summarized back into the main context.
* Agent Teams

  * Context: Each teammate has its own independent context window.
  * Communication: Teammates can message each other directly.
  * Coordination: They share a task list and self-coordinate.
  * Best for: Complex problems that require discussion, iteration, and collaboration.
  * Token cost: Higher — each teammate runs as a separate Claude instance.

Rule of thumb:

* Use subagents for structured delegation.
* Use agent teams when collaboration and shared reasoning are required.

### How to save tokens when using Agent Teams mode?

Agent teams spawn multiple Claude Code instances, each with its own context window. Token usage scales with the number of active teammates and how long each one runs.
To keep agent team costs manageable:
• Use Sonnet for teammates. It balances capability and cost for coordination tasks.
• Keep teams small. Each teammate runs its own context window, so token usage is roughly proportional to team size.
• Keep spawn prompts focused. Teammates load CLAUDE.md, MCP servers, and skills automatically, but everything in the spawn prompt adds to their context from the start.
• Clean up teams when work is done. Active teammates continue consuming tokens even if idle.

## The Ultimate Token saving tips

In LLM systems, token cost grows from:

* Repeated instructions
* Large context injection
* Tool descriptions
* Workflow explanation overhead
* Multi-step delegation chatter

**What Skills Actually Do** (Under the Hood)

Skills solve a very specific systems problem:

> Avoid repeatedly injecting long procedural instructions into the model context.

They do this via lazy capability loading.

With Skills

You define once:
```
.claude/skills/python-refactor/skill.md
```
Now the workflow becomes:
```
User: Use python-refactor skill on module X.
```
Internally:
- Parent agent loads skill definition
- Only relevant parts are injected
- Execution logic is reused

Now you’re paying a small invocation cost instead of full procedural repetition

**KEY OPTIMISATION TRICK**

If you repeatedly use an MCP tool: **Convert recurring patterns into local skills**.

*How?* Let your agent/model use MCP tool for the first time. Ask it to create a skill.md file with all tool information in there. For all subsequent calls use your skill and not MCP.

*Why?*
Skills:
- Are token-cheaper
- Load lazily
- Avoid repeated overhead

This is cost engineering.

### Where Token Savings Actually Occur

Savings happen in three places:
1️⃣ Instruction Reuse
Large instruction blocks are not re-sent each time. Skill logic is stored in repo — not repeated in chat.

2️⃣ Reduced Cognitive Overhead in Parent Agent
When using skills:
- Parent agent doesn't need to reason from scratch
- It doesn’t decompose workflow every time
- It doesn’t invent procedure repeatedly

That reduces reasoning tokens.

3️⃣ Tool Description Minimization
Tool usage is embedded in skill definition.

## FINAL DOs and DONTs (MUST KNOW)

### ✅ DOs
- Define Clear Rules in CLAUDE.md: Ambiguity increases hallucination risk.
- Use Skills for Repeated Workflows: Avoid repeated prompt inflation.
- Use Plan Mode for Complex Changes: Prevents chaotic refactors.
- Use Scripts for Deterministic Operations: LLMs shouldn’t handle precision rewrites alone. 
- Keep Sub-Agent Context Narrow: Small context = better reasoning.
- Separate Global and Project Concerns: Avoid mixing preferences with architecture.

### ❌ DON’T
- Don’t Load Everything Into Context: Bigger context ≠ better results.
- Don’t Skip Planning on Large Refactors: That’s where most breakage happens.
- Don’t Overuse Global Memory: Project-specific constraints belong in .claude/.
- Don’t Give Sub-Agents Full Repo Access: They should be specialized.
- Don’t Use LLMs for Deterministic Transformations: Use scripts instead.
